# ADR-006: Workspace and Crate Structure

**Status**: Accepted
**Date**: 2026-03-01
**Deciders**: AutomatosX Team (DEFAI Private Limited)

---

## Context

ax-engine has 5 crates: `ax-core` (inference + serving mixed), `ax-metal`, `ax-shim`,
`ax-cli`, `ax-bench`. A key design flaw is that `ax-core` contains both the inference
engine and the serving layer ‚Äî they are entangled via shared types.

ax-serving starts fresh and can establish a cleaner boundary from the start.

---

## Decision

ax-serving uses a 5-crate workspace with clear, non-overlapping responsibilities.

### Crate Map


### Dependency Graph (strict, no cycles)

```
ax-serving-cli   ‚Üí ax-serving-api ‚Üí ax-serving-engine ‚Üí mistralrs-core
ax-serving-shim  ‚Üí ax-serving-engine
ax-serving-bench ‚Üí ax-serving-api
```

No crate may introduce a dependency cycle. `ax-serving-engine` must not depend on
`ax-serving-api`. `ax-serving-api` must not depend on `ax-serving-shim`.

### Crate Responsibilities

**ax-serving-engine**
- `InferenceBackend` trait (the only public interface)
- `MistralrsBackend` struct implementing the trait
- `ThermalMonitor` ‚Äî macOS thermal pressure polling
- `MemoryBudget` ‚Äî sysctl-based available-memory check at load time
- `LoadConfig`, `ModelHandle` ‚Äî opaque types for model lifecycle

**ax-serving-api**
- `ServingLayer` ‚Äî shared state (registry, metrics, paged KV, backend reference)
- `ModelRegistry` ‚Äî thread-safe model lifecycle (Arc<RwLock<HashMap>>)
- gRPC server (`grpc/service.rs`) ‚Äî tonic EngineService implementation
- REST server (`rest/routes.rs`) ‚Äî Axum OpenAI-compatible routes
- `MetricsStore` ‚Äî InferenceMetrics, LatencyHistogram, RSS tracking
- `PagedKvManager` ‚Äî Phase 2: prefix cache (ported from ax-engine)
- No inference logic; delegates everything to `InferenceBackend`

**ax-serving-shim**
- `cdylib` crate type ‚Üí builds `libllama.dylib`
- 14 `#[no_mangle] extern "C"` functions matching llama.h
- Opaque `LlamaModel` and `LlamaContext` handles
- All unsafe pointer operations confined to this crate
- Delegates to `ax-serving-engine::InferenceBackend`
- Header generated by `cbindgen`; never hand-edited

**ax-serving-cli**
- Two binaries: `ax-serving` and `ax-serving-api`
- `ax-serving`: llama.cpp-compatible flags via clap; default mode = single inference; `serve` subcommand starts a worker node (REST + gRPC) via `ax-serving-api`
- `ax-serving-api`: multi-worker API gateway (orchestrator); no model weights, pure dispatch

**ax-serving-bench**
- Binary crate: `ax-serving-bench`
- Subcommands: `bench`, `soak`, `profile`
- Ported from ax-engine `ax-bench` with backend reference updated
- JSON output for CI regression detection

## Platform Gate

Every crate has a compile-time platform check:

```rust
// Top of lib.rs / main.rs in every crate
#[cfg(not(all(target_arch = "aarch64", target_os = "macos")))]
compile_error!("ax-serving only supports aarch64-apple-darwin (Apple Silicon M3+)");
```

## Naming Convention

| ax-engine name | ax-serving name | Reason |
|---|---|---|
| `AX_*` env vars | `AXS_*` env vars | Avoid collision in dual-install environments |
| `ax-engine.sock` | `ax-serving.sock` | Separate UDS namespace |
| `EngineService` (proto) | `ServingService` (proto) | Rename for clarity |
| `LlamaModel` (C API) | `LlamaModel` (unchanged) | ABI compatibility requires same name |

## Consequences

### Positive

- Clean inference/serving boundary prevents the entanglement that plagued ax-core
- `ax-serving-shim` is completely isolated; all unsafe is in one crate
- Benchmark crate depends only on the public API, same as external users
- Adding a new backend (e.g., future CUDA) requires only a new `InferenceBackend` impl

### Negative

- 5 crates means 5 `Cargo.toml` files; workspace-level dependencies mitigate duplication
- `ax-serving-api` is still moderately large (~2000 lines); could split into
  `ax-serving-grpc` and `ax-serving-rest` in future
