# AX Serving Progress Report

Date: 2026-03-25

## Summary

This report captures the current progress on bug fixing, serving-path performance work, validation, and benchmark findings.

## Completed Fixes and Improvements

### Bug fixes

- Fixed a `llamacpp` closure / `?` compile issue.
- Added `mistral` chat-template compatibility rendering so chat requests no longer fail on that model family.

### Streaming and hot-path improvements

- Batched steady-state token emission in the native backends while keeping first-token emission immediate.
- Added `AXS_STREAM_TOKEN_BATCH` with default `4` and clamp `1..=32`.
- Removed allocation-heavy dispatch selection paths in orchestration policy code.
- Bounded unhealthy-worker TCP probe concurrency in the health ticker.
- Reused existing percentile snapshots in the Prometheus route instead of recomputing them.
- Added a lazy sorted cache for latency histogram percentiles.
- Reduced dispatch-path cloning by:
  - pushing preferred-pool filtering and reroute exclusion into registry selection
  - slimming `WorkerStatus` to dispatch-relevant fields
  - moving inflight counter lookup to the selected worker only
  - adding a fast path for requests without preferred-pool routing
  - caching `AXS_ROUTING_TRACE` with `OnceLock`

### Admission-capacity fix

- Identified a capacity mismatch:
  - worker-advertised `max_inflight` defaulted to `8`
  - serving-layer default per-model limit was `4`
- Fixed worker registration so advertised `max_inflight`:
  - defaults to `min(sched_max_inflight, sched_per_model_max_inflight)`
  - is clamped to actual scheduler limits even if `AXS_WORKER_MAX_INFLIGHT` is set higher

## Benchmark Work Completed

### Direct worker vs orchestrator compare

Two direct-vs-orchestrator compare runs were executed against a local worker and local orchestrator using the same model.

Artifacts:

- `target/bench-compare-routing-direct-vs-orchestrator.json`
- `target/bench-compare-routing-orchestrator-vs-direct-reversed.json`

Key finding:

- Once warm-state bias was controlled by reversing order, direct worker and orchestrator were effectively tied.
- Routing overhead in this single-worker setup is in the noise floor.

### Multi-worker benchmark against orchestrator

Artifacts:

- `target/bench-multi-worker-routing.json`
- `target/bench-multi-worker-routing-workers4.json`
- `target/bench-multi-worker-routing-workers8-after-capacity-clamp.json`

Key findings:

- `workers=8` initially produced heavy `503` shedding.
- `workers=4` was stable and clean:
  - `60/60` success
  - `P50 ~= 1428 ms`
  - `P95 ~= 1456 ms`
  - `P99 ~= 1456 ms`
  - `throughput ~= 2.8 req/s`
- After the advertised-capacity clamp fix, the `workers=8` benchmark improved only slightly:
  - before: `19` success / `41` errors
  - after: `23` success / `37` errors

## Current Technical Conclusion

- The dispatch-path optimizations appear correct and low-risk.
- Single-worker routing overhead is not the dominant bottleneck.
- The remaining `workers=8` overload behavior is not just a registration mismatch.
- The direct orchestrator path still hard-fails with `503` when all registered workers are at capacity.
- With one worker and real effective capacity `4`, `workers=8` remains beyond what the current direct-mode admission behavior will sustain without shedding.

## Current Best Next Step

The next engineering decision is architectural rather than micro-optimizing the same path again.

Recommended next actions:

1. Add a second worker and rerun the multi-worker benchmark to validate real scale-out behavior.
2. If single-worker `workers=8` must not shed requests, implement explicit orchestrator-side queueing/backpressure instead of direct hard-fail admission.
3. Move to higher-upside throughput work such as real generation batching once the admission behavior decision is made.

## Operational Note

Local benchmark worker and orchestrator processes were started and cleaned up during testing. Leftover `ax-serving` and `llama-server` processes were explicitly terminated after the benchmark runs.
